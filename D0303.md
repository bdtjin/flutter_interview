# RESTful API란 무엇인지 REST의 핵심 원칙에 맞추어 설명하세요.

## REST
REST (Representational State Transfer)는 웹의 기존 기술과 HTTP 프로토콜을 그대로 활용하기 위한 아키텍처 스타일입니다.

## RESTful API
RESTful API란 이 REST 원칙을 철저히 따라 설계된 API를 말하며, 자원을 이름으로 구분하여 그 상태를 주고받는 소통 방식을 의미한다.

## REST의 핵심 원칙

1. 자원 기반
모든 것을 자원으로 간주하고, 각 자원에는 고유한 URI를 부여한다.
- 예시. /users, /posts, /comments
- 잘못된 예시. /getUsers, /postPosts, /commentComments

2. 무상태성
서버는 클라이언트의 이전 상태를 저장하지 않는다. 각 요청은 독립적이며, 요청에 필요한 모든 정보를 담고 있어야 한다.
- 장점: 서버의 부담이 줄어들고 확장성이 매우 높다.

3. 캐시 가능성
HTTP라는 기존 웹 표준을 그대로 사용하기 때문에, 웹의 강력한 기능인 캐싱을 적용할 수 있다.
- 효과: 대량의 요청을 효율덕으로 처리하고 응답 속도를 높일 수 있다.

4. 자체 표현 구조
REST API 메시지만 보고도 이 요청이 무엇을 의미하는지, 어떤 데이터를 담고 있는지 직관적으로 알 수 있어야 한다.
- 예시. `GET /users`는 "사용자 목록을 가져오라"는 뜻임을 누구나 쉽게 이해할 수 있다.

5. 인터페이스 일관성
URI로 지정된 자원에 대한 조작을 통일되고 한정적인 인터페이스로 수행한다. 
자원을 조작할 때 쓰는 도구(HTTP Method)를 통일합니다.

### 핵심
- 모든 데이터는 URI(명사)로 정의합니다.
- 모든 행위는 **표준버튼(HTTP Method)**로 통일합니다.
- 각 요청은 독립적(Stateless)이어야 합니다.

## REST의 구성
- 자원 (Resource)
  - REST에서는 모든 데이터를 이름 (URI)으로 정의합니다.
- 행위 (HTTP Method)
  - GET, POST, PUT, DELETE
- 표현 (Representation 응답 데이터)
  - 주문이 끝나면 상태 코드와 함께 JSON 텍스트 형식으로 데이터를 보내준다.


---

도서관 시스템으로 예시를 들어보자.

| 행위 | HTTP Method | URI | 의미 |
| --- | --- | --- | --- |
| 조회 | GET | /books/10 | 10번 책의 정보를 가져온다. |
| 등록 | POST | /books | 새로운 책을 등록한다. |
| 수정 | PUT | /books/10 | 10번 책의 정보를 수정한다. |
| 삭제 | DELETE | /books/10 | 10번 책의 정보를 삭제한다. |


## 핵심
결론적으로 RESTful API는 HTTP 프로토콜의 장점을 극대화하여 별도의 인프라 구축 없이도 확장성과 유연성이 뛰어난 시스템을 만들기 위한 규약입니다.

Flutter 앱을 개발할 때도 이러한 표준을 준수하면 백엔드 서버와 더 명확하고 효율적으로 통신할 수 있습니다.
